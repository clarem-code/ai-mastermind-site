Session 1: Your First AI Agent
CLAUDE.md · Context Documents · Skill Creator · Add the Egg
▶

Three Foundational Pieces

  • CLAUDE.md — ~100 lines, acts as your AI's employee playbook. Sets core mindset, planning mode behavior, MCP tool configs, file organization rules, and pointers to context documents. Created through natural language — just talk to Claude and tell it what you want it to know.
  • Context Documents — Files Claude reads when it wakes up: Personal Constitution, Team Context, Business Goals, Vendor/Client Lists, Service Descriptions. Stored in a "Master Context Documents" folder.
  • Skill Creator — A meta-skill that lets Claude build new skills for itself. Better than writing skills manually. Install it first, then use it to build everything else.

Add the Egg

The central creative principle of the program: go beyond the minimum viable output. A morning briefing could just list your calendar — but add weather, goal-alignment, quick tasks Claude assigns itself, delegation suggestions, local events, and a motivational quote. The "egg" is the extra feature that turns useful into delightful.

Morning Briefing Example (built during Session 1)

  • Date and day of week with weather forecast
  • Energy readout and emoji-based mood system
  • Calendar summary with location details and prep notes
  • Context from goals document — "is today goal-aligned?"
  • Claude-assigned quick tasks for the day
  • Delegation suggestions for the team
  • Local events and fun facts
  • Motivational quote

MCP (Model Context Protocol)

The universal connection standard between Claude and external tools (Gmail, Calendar, Slack, Notion, etc.). Cowork uses "Connectors" — toggle-based, no setup. Claude Code requires a bit more configuration but can be set up through natural language.

Scheduling Options Introduced

  • /schedule — Cloud-based, runs while you sleep, no local file access, minimum 1 hour. Recommended first choice.
  • Cron jobs — Local execution, requires computer to be on, full file access, minimum 1 minute.
  • Cowork scheduled tasks — UI-friendly, reliable, good for most users.
Get Fully Operational
Terminal setup · File management · Permissions · Voice dictation
▶

What Was Covered

  • Installing Claude Code on Mac (Homebrew) and Windows
  • Terminal vs. iTerm2 (split windows, better UI) vs. VS Code terminal
  • Shell aliases: shortcut commands like c for claude --chrome, stored in .zshrc
  • Flags: --chrome for browser access, --enable-auto-mode for smart permissions
  • File management: controlling where Claude saves files, naming conventions, folder structure to prevent desktop clutter

Voice Dictation

Wispr Flow (preferred): hold the function key, speak, release. 4x faster than typing. Used 50–150 times per day by power users. Skip the built-in /voice command — Wispr Flow is significantly better.

Permission Levels

  • Default (Ask Everything) — Most conservative. Approves every action. Leads to decision fatigue but good for learning.
  • Allow List — Pre-approve common safe actions. Read files: always allow. Delete files: always ask.
  • Auto Mode — "Santa mode": a mini model classifies actions as risky or safe. Asks only for genuinely risky ones. Best for experienced users.
  • YOLO Mode — Skips all safety checks. Only use inside a Docker sandbox, never on real work.

What Claude Can Do

  • Read and write files, create and organize folders
  • Edit documents in place, handle images and PDFs
  • Access browser with Chrome extension (/ss for screenshots)
  • Run code, access APIs, connect to MCPs
  • Computer use (control mouse and keyboard — use carefully)
Session 2: AI Agent Operating System
File systems · Privacy · Memory management · Progressive trust
▶

How Claude Wakes Up (Context Stack)

  • Model base personality (Opus 4.7, etc.)
  • CLAUDE.md file(s) — stacked, more specific files take priority
  • memory.md file (if exists)
  • Skill summaries (not full contents)
  • Project metadata and folder names
  • Does NOT automatically get: file contents, subfolder files, MCPs not yet called

Project Organization ("Goldilocks Principle")

Projects are dedicated folders for multi-session efforts. Not too broad ("Sales"), not too narrow ("Cold email replies") — find the right scope. Good example: "Enterprise Renewals." Bad: "Sales" (too broad). Access via cd ~/.claude/projects/ProjectName then claude.

Progressive Trust Model

  • Level 1 — Ask Everything: Approves every action. Good for learning, leads to fatigue long-term.
  • Level 2 — Allow List: Build approved actions over time. Gradually reduce friction for safe operations.
  • Level 3 — Auto Mode: Classifies risky vs. safe. Asks only for genuinely risky actions. Enables multi-threading.
  • Level 4 — YOLO: Skip all checks. Only in sandboxed environments for research tasks.

Data Privacy: What Actually Happens

  • File lives on your computer until you input it into Claude
  • Your prompt + file sent to Anthropic's API (Arizona/Wyoming data center)
  • Model processes in cloud, response returns
  • By default, data is NOT used for training (opt-out automatic since 2024)
  • 30-day retention for moderation purposes only
  • Enterprise options: AWS Bedrock or Google Vertex (data never leaves your cloud)

Memory Management

  • Auto Memory — Built-in, default on. Use /memory to view, remember [thing] to save. Still imperfect — treat as supplementary.
  • Manual Updates — More reliable: "Remember that…" commands, updating CLAUDE.md, updating context documents.
  • Claudeopedia — Personal knowledge base using Obsidian. Connects sources (transcripts, articles), creates knowledge graph. Portable to any AI system.

Key Concept

"Files are all you need." — A clean, well-organized file system is the foundation of every powerful AI system.

Technical Office Hours #1
GitHub · API security · Team setup · Tool comparisons

Key Topics

  • GitHub is how you sync skills across devices — version control everything. Update skill → commit → push → pull on another device. Claude writes the commit messages.
  • API key security — Use environment variables (stored in .profile on Mac). Never paste keys in chat. Set monthly billing limits. Reference by name, not value.
  • Claude Code vs. Cowork vs. Web App — Claude Code has all capabilities plus more. Cowork is better for visual people who want artifacts. Web app: only for quick one-off questions.
  • Team setup (4 people) — Use GitHub for skill sharing. One account per person with access to a shared repo. Better rate limits on one coordinated account vs. per-person.
  • Context7.com — API/MCP marketplace. Search for any service; everything packaged for Claude Code.

CLAUDE.md vs. Skill vs. Context Doc: When to Use Which

  • CLAUDE.md — First instructions, orientation for a directory or project
  • Skills — Reusable across projects, carry everywhere
  • Context docs — Reference files, pulled in when needed, not in chat
  • Use reference files in CLAUDE.md to avoid token bloat

Key Lesson

Lower-level tools (Claude Code) are harder to start with but meaningfully more powerful. The investment pays off quickly.

Capstone: AI First Operations
Live CRM build · Connect → Continue → Compound · 3-agent architecture
▶

The Problem

A lean ops team managing hundreds of conversations with no connective tissue across tools — Gmail, Notion, meeting transcripts, Slack, Calendar — and no persistent context between sessions.

The Solution: Custom AI-Powered CRM

Built using Claude Code + Notion + Slack. Three discrete agents with narrow responsibilities replacing one massive agent that had been failing at 72% context compaction.

Three-Agent Architecture

  • Agent 1 — The Bookkeeper (CRM Sync): Runs twice daily. Validates Notion data before and after updates. Ensures data integrity — names, duplicates, status fields.
  • Agent 2 — The Workhorse (Client Intelligence): Gathers from Gmail, calendar, meeting transcripts. Scans in parallel (2-week lookback, 200 results). Plans all Notion updates. Writes clean data.
  • Agent 3 — The Intelligence (Warmth Report): Reads clean data from shared file. Complex analysis: deals at risk, days since last contact, prioritization. Posts daily report to Slack.

Connect → Continue → Compound (The Three-Question Test)

  • Connect? — Does it link multiple tools/data sources? If single source: it's just automation.
  • Continue? — Does it run without you? Can you go on vacation and it keeps working?
  • Compound? — Does it start small and get smarter? Can you add layers without rebuilding?

Best Practices Demonstrated

  • Discrete agents with narrow focus: when something breaks, you know exactly where to look
  • Shared CLAUDE.md: global CRM instructions apply to all three agents
  • Step-by-step execution strips: not just personas, detailed instructions (e.g., "Search Gmail max 200 results, match by domain, write to JSON")
  • Ship early, audit often: these systems degrade without regular attention

Key Lesson

"Build your agents like you'd hire for your best team — clear roles, clear expectations, and no ambiguity about what success looks like."

Session 3: Build Something You Can Sell
Web apps · PRDs · Deployment · Payments · Testing personas
▶

The Build vs. Buy Shift

The old approach: evaluate 3 existing solutions. The new approach: can I build this myself in a weekend? A 50% solution giving 90% of the value — tuned exactly to your needs — often beats any off-the-shelf product.

Four Levels of App Builds

  • Level 1 — Share with a few people: Artifact in Claude (public), Google AI Studio (password-protected), GitHub Team (private). Set up in minutes. Good for internal prototypes.
  • Level 2 — Share with the world: Personal website, interactive tools on Replit or Lovable, embed in existing site. No infrastructure required.
  • Level 3 — Add AI functionality: Claude API calls in-app for real-time generation. Lead gen tools, visualization tools, personalized experiences.
  • Level 4 — Charge for it: Google SSO for logins, Stripe for subscriptions, webhooks for payment confirmation, secrets in Vercel.

Deployment Flow

Code and test locally (localhost) → Push to GitHub (Claude writes the commit) → GitHub auto-triggers Vercel → Live in seconds.

PRD: Write Before You Build

A Product Requirements Document is 200+ lines written before a line of code. It covers: problem statement, user stories, input/output specs, architecture recommendations, and design guidelines. It aligns you with Claude before building, prevents wasted effort, and dramatically improves output quality.

Four Personas Testing Framework

  • The Creator (you) — Usage analytics, daily spend cap, review logs
  • The Happy User — Easy sharing, clear prompts, error-free flows
  • The Accidental User — Disclaimers, guidance, helpful error messages
  • The Adversary — Prompt injection protection, input length limits, API abuse prevention

API Key Security Rules

  • Set a hard monthly billing limit on your account
  • Store keys in environment variables — never hardcode
  • .env files are excluded from GitHub automatically
  • Store production secrets in Vercel, not in your codebase

Competitive Moat Timing

  • 2 minutes to build = instantly copied, no moat
  • 2–8 hours = defensible moat (the sweet spot)
  • 2+ weeks = significant but high scaling cost
Technical Office Hours #2
VS Code · GitHub best practices · Memory · Responsive design
▶

Key Topics

  • VS Code integration — Better visibility than terminal-only. Red highlights show changed files. Easy to supervise large agent runs. Simple changes are obvious; complex ones benefit from visual diff.
  • Don't trust the built-in memory feature yet — Still an early product, unreliable. Rely on skills + CLAUDE.md for reliable persistence. If you want something remembered: write it to a file in a known location.
  • GitHub best practices — Commit every 2 hours. Let Claude write 400-word commit messages. Easy revert to specific points. Never store git repos in Google Drive — two sync systems fight each other.
  • Responsive design — Claude often lies about mobile responsiveness. Take actual screenshots on device. Flag specific failing elements. Define design guidelines in detail (fonts, colors, breakpoints) before building.
  • Permissions in .claude/settings.json — Pre-configure defaults. "Always allow" button persists across sessions. Always allow: read files, basic commands. Keep gated: edits, deletions.

Sharing Skills Across a Team

Create a GitHub repo with shared skills organized by function (marketing, operations, comms). Each team member forks or clones. Option for a private fork of Claude's repo with all shared skills included. Full team access, version-controlled, easy to audit.

Session 4: Building Your Digital Workforce
Multi-agent systems · Chief of Staff model · Orchestration · Scale
▶

Important Framing

This is about workforce multiplication, not replacement. Agents as infrastructure that extends what a human can do — not a substitute for judgment, strategy, or relationships.

Where Most Users Are

  • 99% of users: assistant mode (back-and-forth chat)
  • Super users: agent mode (complete tasks, take action)
  • The frontier: workforce mode (multiple agents orchestrated together)

Skills vs. Capital-A Agents

  • Skills (lowercase): Inherit global context and permissions. Same model as chat. Sequential execution. Easy to share. Good for most tasks.
  • Agents (capital A): Specify model explicitly. Define tool access. Can run in parallel. Limited context (you control what they see). Better cost control. Better safety boundaries.

Speed Difference

With skills: everything runs sequentially. With agents: 5 parallel agents complete in the time of the slowest one. 20 agents on different subtasks = same clock time as 1. This is a fundamental shift for complex work.

Digital Workforce Evolution

  • Version 1 (Flat Org) — 3 directors, all same model, no coordination, you manage all of them directly. Simple to start but doesn't scale.
  • Version 2 (Hierarchical) — 6 directors, each with specialists. More coordination. More capable.
  • Version 3 (Chief of Staff) — One CoS agent is the only agent you interact with. CoS orchestrates 6 directors. Historian agent embedded with CoS for continuous memory. Civilian agents spun up for specific tasks. Much better delegation.

Common Mistakes

  • Using Opus for everything — slow and expensive. Use Sonnet/Haiku for non-critical tasks.
  • Going YOLO mode on complex multi-agent systems — dangerous
  • Under-investing in orchestration, memory, and observability
  • One massive agent instead of discrete focused ones
  • Using real people's names for agents — causes confusion

Key Insight

The best companies prioritizing AI are winning not on model quality but on speed of iteration. Teams of 2–6 with strong AI infrastructure are outcompeting teams 10x their size.

Core Frameworks

Six Frameworks That Change How You Work

These are the ideas I kept coming back to. Not technical concepts — thinking patterns. The difference between people who get real value from AI and people who don't usually comes down to whether they've internalized something like these.

FRAMEWORK 01
Build Your Foundation
The three pieces every AI system needs before anything else works.

Without a foundation, every Claude session starts from scratch. With one, your AI acts like a trained employee who already knows your context, your priorities, and how you like to work.

  • CLAUDE.md — Your AI's employee playbook. ~100 lines. Sets the core mindset (e.g., "think like my chief of staff"), planning mode instructions, MCP tool configs, file organization rules, and pointers to your context documents. Create it through natural language: just tell Claude what you want it to know.
  • Context Documents — What Claude needs to know about you: your role, your goals, your team, your vendors, your service lines. Stored in a "Master Context Documents" folder. Updated through conversation. Referenced in CLAUDE.md so they load without you having to paste them every session.
  • Skill Creator — A meta-skill that lets Claude build new skills for itself. Better than writing skills manually. Install this first, then use it to build every other skill you need.
Without CLAUDE.md and context documents, you're starting every session explaining who you are. With them, Claude already knows — and it remembers what you care about.
FRAMEWORK 02
Add the Egg
Go beyond the minimum. Every workflow should have delight built in.

The principle: never ship a workflow that just does the minimum. Ask, "What if I added X to make this even better?" Each addition is small, but the compound effect turns something functional into something people actually want to use.

Example: A morning briefing could just list your calendar. But add weather, an energy/mood score, goal-alignment context, Claude-assigned quick tasks, delegation suggestions, local events for the day, and a closing motivational quote — and it becomes the first thing you open every morning.

  • Start with the minimum viable version
  • Test it with one real user (yourself)
  • After each use, ask: "What's the one thing missing?"
  • Add that one thing. Test again. Repeat.
The instructor built 10+ versions of her morning briefing before it felt right. That's not failure — that's the whole point. The willingness to keep iterating is what separates people who build tools they actually use from people who have nice demos.
FRAMEWORK 03
Builder Framework
Start with the problem. Design the solution. Map the workflow. Add the egg.

The most common mistake new builders make: starting with the technology. The Builder Framework forces you to start with the problem instead.

  • Step 1: Define the problem — What friction exists? What takes too long, gets dropped, or requires manual effort every time?
  • Step 2: Design the solution — What would it look like if that friction disappeared? Ignore implementation for now.
  • Step 3: Map the workflow — What triggers it? What does it do? What's the output? Who receives it?
  • Step 4: Add the egg — Now that the baseline is defined, what additional feature would make this genuinely delightful?

Then go into plan mode with Claude, walk through the four steps, and build. Don't build first and plan later.

Technology for its own sake is the fastest path to a tool no one uses. Start with the real problem. Build the simplest version that solves it. Then make it remarkable.
FRAMEWORK 04
Progressive Trust Model
Build AI confidence in layers — not all at once.

Giving an AI system full autonomy on day one is like handing your car keys to someone you just met. The Progressive Trust Model is a governance framework for expanding AI access responsibly.

  • Level 1 — Ask Everything: Claude requests approval before every action. Conservative, exhausting, but excellent for learning what the system actually does.
  • Level 2 — Allow List: Pre-approve common low-risk actions (read files, search the web). Keep gates on high-risk actions (delete files, push to production). Build this list iteratively.
  • Level 3 — Auto Mode: A lightweight model classifies actions as risky or safe. Claude handles safe actions autonomously and asks only for the risky ones. Enables multi-threading without decision fatigue.
  • Level 4 — Sandboxed YOLO: Full autonomy within an isolated environment (Docker container). No real data. For experiments, research, and overnight tasks. Never on production systems.
The goal isn't maximum autonomy. It's the right level of trust for each situation. Move deliberately. Expand access as you earn confidence — in the system and in yourself.
FRAMEWORK 05
Connect → Continue → Compound
The three-question test for real AI workflows vs. dressed-up automation.

Before investing in any AI system, run it through these three questions:

  • Connect? — Does it link multiple tools or data sources together? An AI that only touches one system is just automation with better text. Value comes from the connections between systems.
  • Continue? — Does it keep running without you present? Can you go on vacation and the system still works? If it stops when you stop, it's a detractor from your workday, not an addition.
  • Compound? — Does it start small and get smarter? Can you add layers without rebuilding? You want a system designed for today, with room to grow tomorrow.
If your workflow answers yes to all three, you have a real system — one that compounds over time. If it fails one, you now know exactly what's missing and what to build next.
FRAMEWORK 06
Learn → Adapt → Make It Yours
The fastest path from "I saw something that works" to "I have something that works for me."

The creative shortcut that accelerates every build: you don't have to invent from scratch, and you shouldn't copy blindly. The three-step process:

  • Learn: Screenshot, save, or note anything that works well — a workflow, a prompt structure, an interface pattern, a skill template. Build a collection.
  • Adapt: Feed it to Claude with context about your situation. "Here's what this person built. Here's how my situation is different. What should we change?"
  • Make It Yours: Take the adapted version, run it, iterate on it. The result will be tuned to your exact context, team, and goals — not a generic template.

Less than 20 words of prompting often needed. Faster than building from scratch. Smarter than copying directly.

The best builders aren't the most original. They're the fastest at seeing what works, understanding why, and making it their own. That's a learnable skill — and it compounds.

Building Your Digital Workforce

The Chief of Staff Model

Building a digital workforce means thinking about AI the same way you'd think about building a great human team: how many people can you realistically manage directly, what distinct roles do you need, and what skill sets fill the gaps? A well-structured agent hierarchy has clear roles, clear reporting lines, and a single point of contact — so you never have to remember who does what or manage everyone individually. Chief of Staff

The only agent you talk to directly. Receives outcomes from you, decides which Directors to involve, sequences the work, synthesizes results. Thinks ahead: "What's the best version of what you'd want — not just the literal request?"

The Historian

Solves the hardest problem in AI systems: memory. Always running alongside the CoS. Continuously takes notes, documents decisions, captures key context. No more losing thread between sessions. Institutional memory that persists.

Directors

Each owns one functional lane — communications, research, operations, etc. The CoS delegates to Directors; Directors manage their own Specialists. Directors never reach across each other's lanes. Clear accountability, no overlap.

Civilians

Temporary, task-specific agents. Spun up by Directors for a single focused task, then dissolved. Ephemeral by design. Use the cheapest model (Haiku). Don't accumulate — create and dissolve.

Model Tiering

Cost control through deliberate model selection. Opus (most capable, most expensive): CoS only. Sonnet (balanced): Directors. Haiku (fast, cheap): Specialists and Civilians. Don't use Opus everywhere — it's slow and costly for routine tasks.

Lane Discipline

The rule that makes orchestration work: Directors own their lane and never reach into another Director's team. When two lanes overlap, the CoS handles coordination — not the Directors directly. Codified in each Director's instructions.

"Build your agents like you'd hire for your best team — clear roles, clear expectations, and no ambiguity about what success looks like."

Claude's command-line interface. More powerful than the web app: persistent file access, local execution, full MCP connectivity, project memory, and the ability to run scheduled tasks overnight while you sleep.

Multi-Agent Systems

▶

Skills vs. Capital-A Agents

  • Skills: inherit all global context and permissions, same model as chat, run sequentially, easy to share across projects. Good for most tasks.
  • Agents: you specify the model (Opus/Sonnet/Haiku), define tool access explicitly, can run in parallel, receive only the context you give them. Better cost control, better safety isolation.

The speed difference is significant: 5 agents in parallel complete in the time of the slowest one, not the sum of all five.

Why One Big Agent Fails

  • Context compaction: loses track of what it was doing mid-task
  • Unclear failure points: when it breaks, you don't know where
  • Slow: everything sequential, no parallelism
  • Hard to update: change one thing, everything else shifts

The Fix: Discrete Focused Agents

  • One agent = one narrow responsibility
  • Agents pass clean outputs to each other via shared files
  • When something breaks, you know which agent to fix
  • Each agent can be updated without affecting others

Sub-Agents and Hooks

  • Sub-agents: Agents that spin up autonomously within a larger task when the scope grows too big. Claude creates these automatically when needed, or you can specify them.
  • Hooks: "Before X happens, do Y" or "After X completes, do Z." Automated logic attached to events in your system.

Agent Teams Feature

Built-in Claude Code feature: multiple agents share files and pass notes back and forth. Token-heavy and slow. Use sparingly (once a week for major complex problems). Great for: getting multiple "perspectives" on a strategy, running parallel research on different angles, board-room-style debates.

Implications for Training

A Framework for Enterprise AI Adoption

This is my recommendation, based on my experience with this program. I'm sharing what I believe would genuinely move the needle at the company — for a Microsoft 365 environment with a non-technical audience.

1

Get Leadership Buy-In — Starting with a Live Demo

Don't pitch AI with slides. Show it working on something real, in real time, in the room.

The trick is to show it — not explain it.

What to demo:

  • An AI agent completing an end-to-end task — researching a topic, drafting a document, summarizing a meeting, organizing a to-do list — without being micromanaged step by step
  • Microsoft Copilot embedded in Outlook or Teams — familiar environment, immediate relevance
  • A "what would disappear from your plate?" moment: ask each leader directly, then show Claude handling a version of it live

Key framing for leadership:

  • This is not a chatbot. It's a system that operates on your behalf, even when you're not watching.
  • The competitive advantage isn't the AI model itself — it's the speed of iteration of the team using it.
  • Teams of 5 with AI infrastructure are already outcompeting teams of 50 without it. This is the window.
2

Train the First Cohort: Directors & Managers

Why start here: Directors and managers have enough organizational context to identify high-value use cases. They set the tone for their teams. Their wins become the case studies that drive the next cohort. Bottom-up adoption is slower and harder to sustain.

Training structure (modeled on this mastermind program):

  • Biweekly formal sessions — 60–90 minutes, cohort-based, hands-on. Each session ends with a concrete deliverable to build before the next one.
  • Office hours on alternating weeks — drop-in, no agenda. Participants bring what broke, what they built, what they're stuck on.

Suggested Microsoft Teams community infrastructure:

#announcements
Session schedule, recordings, updates. Admin-only posts.
#ask-the-community
Peer-to-peer questions. "How did you get Claude to…?" Anyone can answer.
#show-and-tell
Share what you built, what's working, what surprised you. Builds culture.
#admin-support
Technical help, account issues, permission problems. Staffed by training leads.

Suggested session arc (5 sessions + office hours):

  • What is Claude? Live demo + writing your first CLAUDE.md
  • Building context documents and your first automated skill
  • Scheduling: running tasks overnight without being present
  • Building workflows that connect your existing tools
  • Introduction to multi-agent thinking: when one agent isn't enough

Office hours run between each session. The program is designed to be iterative — assess after each session, adjust before the next.

Iterative by design:

  • Expect version 1 of everything to be imperfect. That's not a failure — that's the process.
  • Build "bragging about how bad v1 was" into the culture. Progress is the metric, not polish.
  • Review after every two sessions: what's working? What isn't? What do participants actually need next?

Outstanding Questions Before Launch

  • How large should the first cohort be? (Recommended: 10–15 for high engagement and accountability)
  • Who leads the training? Internal champion vs. external facilitator vs. hybrid?
  • What does success look like at 30, 60, and 90 days?
  • How does this integrate with existing L&D programs and compliance requirements?
  • What data handling guidelines apply to Claude usage in this environment?
